# Spec: Run Model Elevation & API Contract Cleanup

**Status:** Approved for implementation
**Date:** 2026-03-20
**CTO review issues addressed:** #1 (Result Delivery), #2 (GET /jobs/{job_id} scope), #4 (Run Model Elevation)
**Out of scope:** Issue #3 (Test coverage / CI harness) — separate work order.

---

## 1. Motivation

Scrapeyard is approaching volume usage against optics retailers for the EyeBox
project. The current data model has three problems that become acute at scale:

1. **Run identity is fragmented.** `jobs.current_run_id` tracks only the latest
   run. `results_meta.run_id` ties results to runs. But `errors` has no
   `run_id` column — errors cannot be associated with a specific run. For
   scheduled jobs that fire frequently, this makes debugging extraction failures
   ambiguous.

2. **The `/jobs/{job_id}` endpoint is too thin.** It returns a flat job summary
   with no config, no run history, and no next-scheduled-run info. Consumers
   must make multiple calls and cross-reference to understand job health.

3. **The output format system is overbuilt for the actual consumer.** The sole
   active consumer (EyeBox) needs JSON. The `html`, `markdown`, and
   `json+markdown` formats, the `formatters/` module, and the `OutputFormat`
   enum exist to serve use cases that no longer apply (job-scout is
   discontinued). This dead code adds surface area for bugs and cognitive load.

This spec addresses all three by promoting "run" to a first-class persisted
model, expanding the job detail endpoint, and stripping the output format system
down to JSON-only.

## 2. Schema Changes

All changes target a **fresh database**. Existing DBs will be nuked before
deployment. No backfill or ALTER migrations are required.

### 2.1 New table: `job_runs` (in `jobs.db`)

Migration file: `sql/004_create_job_runs.sql`

```sql
CREATE TABLE IF NOT EXISTS job_runs (
    run_id        TEXT PRIMARY KEY,
    job_id        TEXT NOT NULL,
    status        TEXT NOT NULL DEFAULT 'running',
    trigger       TEXT NOT NULL,
    config_hash   TEXT NOT NULL,
    started_at    TEXT NOT NULL,
    completed_at  TEXT,
    record_count  INTEGER,
    error_count   INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX IF NOT EXISTS idx_job_runs_job_id ON job_runs (job_id);
CREATE INDEX IF NOT EXISTS idx_job_runs_started_at ON job_runs (started_at);
```

Column details:

| Column | Type | Constraint | Description |
|---|---|---|---|
| `run_id` | TEXT | PRIMARY KEY | Same value generated by `generate_run_id()`, passed through the enqueue path. |
| `job_id` | TEXT | NOT NULL | FK to `jobs.job_id` (not enforced by SQLite FK constraint; matches existing pattern). |
| `status` | TEXT | NOT NULL DEFAULT 'running' | One of: `running`, `complete`, `partial`, `failed`. Uses same `JobStatus` values as jobs. |
| `trigger` | TEXT | NOT NULL | One of: `adhoc`, `scheduled`, `manual`. Indicates why this run was initiated. |
| `config_hash` | TEXT | NOT NULL | SHA-256 hex digest of the raw `config_yaml` string. Computed as `hashlib.sha256(config_yaml.encode()).hexdigest()`. |
| `started_at` | TEXT | NOT NULL | ISO 8601 UTC timestamp. Set when the worker creates the run row. |
| `completed_at` | TEXT | nullable | ISO 8601 UTC timestamp. Set when the worker finalizes the run (success or failure). |
| `record_count` | INTEGER | nullable | Total extracted records. Set at completion. Null if the run failed before producing results. |
| `error_count` | INTEGER | NOT NULL DEFAULT 0 | Count of errors for this run. Populated at completion via `SELECT COUNT(*) FROM errors WHERE run_id = ?`. |

## 3. Output Format Simplification

### 3.1 Formats removed

| Format | Disposition |
|---|---|
| `html` | Removed. Never worked as spec'd (returned stringified extraction data, not raw page HTML). No consumer. |
| `markdown` | Removed. Existed for the job-scout LLM context window use case. job-scout is discontinued. |
| `json+markdown` | Removed. Same rationale as markdown. |

### 3.2 Surviving format

`json` is the only output format. It is no longer configurable — all jobs
produce JSON output.

### 3.3 Files and code to remove

| Path | Action |
|---|---|
| `src/scrapeyard/formatters/` | Delete entire directory (all 4 files: `__init__.py`, `factory.py`, `html_fmt.py`, `json_fmt.py`, `markdown_fmt.py`). |
| `config/schema.py` — `OutputFormat` enum | Delete. |
| `config/schema.py` — `OutputConfig` model | Replace with a model containing only `group_by`. See section 3.4. |
| `config/schema.py` — `ScrapeConfig.output` field | Update type to new model. |
| `queue/worker.py` — `_OUTPUT_FORMAT_TO_SAVE` mapping | Delete. |
| `queue/worker.py` — formatter imports and dispatch | Replace with inline JSON assembly. See section 5.4. |
| `storage/result_store.py` — `_FORMAT_FILES` mapping | Delete. Always write `results.json`. |
| `storage/result_store.py` — `format` parameter on `save_result()` | Remove. |
| `storage/result_store.py` — format-based file reading in `get_result()` | Simplify to always read `results.json`. |

### 3.4 Surviving config structure

The `output` block in YAML configs retains `group_by` only:

```yaml
output:
  group_by: target  # or "merge" — default: "target"
```

Updated `OutputConfig` model:

```python
class OutputConfig(BaseModel):
    """Output grouping settings."""
    group_by: GroupBy = Field(default=GroupBy.target, description="Result grouping strategy")
```

The `GroupBy` enum (`target`, `merge`) is unchanged.

### 3.5 Grouping logic relocation

The JSON grouping logic currently in `formatters/json_fmt.py` (`format_json`
function) must be moved into the worker's result assembly code. The worker
already builds `formatted_results` and `job_meta` — the grouping (by-target
dict vs flat merged list) is applied to `formatted_results` before passing to
`result_store.save_result()`.

The grouping logic is:

- **`target` (default):** Results keyed by target URL. Each key maps to
  `{"status": str, "count": int, "data": list}`.
- **`merge`:** Flat list of all records with `_source` field injected from the
  target URL.

The worker reads `config.output.group_by` and applies the appropriate
transformation before saving.

## 4. API Contract Changes

### 4.1 `GET /jobs/{job_id}` — expanded detail response

**Current response:**

```json
{
  "job_id": "abc-123",
  "project": "eyebox",
  "name": "brownells-optics",
  "status": "complete",
  "created_at": "2026-03-20T06:00:00Z",
  "updated_at": "2026-03-20T06:05:00Z",
  "schedule_cron": "0 */6 * * *",
  "schedule_enabled": true,
  "last_run_at": "2026-03-20T06:05:00Z",
  "run_count": 12
}
```

**New response:**

```json
{
  "job_id": "abc-123",
  "project": "eyebox",
  "name": "brownells-optics",
  "status": "complete",
  "config_yaml": "project: eyebox\nname: brownells-optics\n...",
  "created_at": "2026-03-20T06:00:00Z",
  "updated_at": "2026-03-20T06:05:00Z",
  "schedule_cron": "0 */6 * * *",
  "schedule_enabled": true,
  "next_run_at": "2026-03-20T12:00:00Z",
  "run_count": 12,
  "last_run_at": "2026-03-20T06:05:00Z",
  "runs": [
    {
      "run_id": "run-20260320-060000-a1b2",
      "status": "complete",
      "trigger": "scheduled",
      "config_hash": "e3b0c44298fc1c14...",
      "started_at": "2026-03-20T06:00:12Z",
      "completed_at": "2026-03-20T06:04:47Z",
      "record_count": 83,
      "error_count": 0
    }
  ]
}
```

**New fields:**

| Field | Source | Notes |
|---|---|---|
| `config_yaml` | `jobs.config_yaml` | Raw YAML string. Already stored on the job row. |
| `next_run_at` | APScheduler in-memory state | Null if job has no schedule or scheduler hasn't registered it. See section 4.1.1. |
| `run_count` | `SELECT COUNT(*) FROM job_runs WHERE job_id = ?` | Derived, not stored on job row. |
| `last_run_at` | `SELECT MAX(started_at) FROM job_runs WHERE job_id = ?` | Derived, not stored on job row. Null if no runs exist. |
| `runs` | `SELECT * FROM job_runs WHERE job_id = ? ORDER BY started_at DESC LIMIT 10` | Last 10 runs, newest first. |

**Removed fields:** None from the response — `run_count` and `last_run_at`
are still present but now derived from `job_runs` instead of denormalized
columns on the `jobs` table.

#### 4.1.1 `next_run_at` implementation

Add a method to `SchedulerService`:

```python
def get_next_run_time(self, job_id: str) -> datetime | None:
    """Return the next scheduled fire time, or None if not scheduled."""
    aps_job = self._scheduler.get_job(job_id)
    return aps_job.next_run_time if aps_job else None
```

The route handler calls this method and includes the result in the response.
If the scheduler has no record of the job (e.g., non-scheduled job, or
scheduler hasn't restarted yet), the field is null.

The route handler for `GET /jobs/{job_id}` gains a dependency on
`SchedulerService` (via `get_scheduler`).

### 4.2 `GET /jobs` — list response with derived run stats

The list endpoint includes `run_count` and `last_run_at` derived via a LEFT
JOIN on `job_runs`:

```sql
SELECT j.*,
       COUNT(r.run_id) AS run_count,
       MAX(r.started_at) AS last_run_at
FROM jobs j
LEFT JOIN job_runs r ON j.job_id = r.job_id
GROUP BY j.job_id
```

The list response shape per job:

```json
{
  "job_id": "abc-123",
  "project": "eyebox",
  "name": "brownells-optics",
  "status": "complete",
  "created_at": "2026-03-20T06:00:00Z",
  "updated_at": "2026-03-20T06:05:00Z",
  "schedule_cron": "0 */6 * * *",
  "schedule_enabled": true,
  "run_count": 12,
  "last_run_at": "2026-03-20T06:05:00Z"
}

```

The list response does **not** include `config_yaml`, `next_run_at`, or `runs`.
Those are detail-only fields.

### 4.3 `GET /results/{job_id}` — updated response envelope

**Current response:**

```json
{
  "job_id": "abc-123",
  "status": "complete",
  "results": { }
}
```

**New response:**

```json
{
  "job_id": "abc-123",
  "run_id": "run-20260320-060000-a1b2",
  "status": "complete",
  "results": { }
}
```

**Changes:**

| Change | Detail |
|---|---|
| Added `run_id` | Echoes which run produced the returned results. Sourced from `results_meta.run_id`. |
| Removed `format` concept | Always JSON. No format field needed. |

`get_result()` in `result_store.py` must return the `run_id` alongside the
result data so the route handler can include it in the envelope. Change the
return type from `Any` to a structured object or tuple containing both.

**Error responses unchanged:**

- 202 if job is still running/queued
- 404 if job not found or no results exist (results endpoint is for data only)
- 400 if `latest=false` without `run_id`

### 4.4 `GET /errors` — no response shape change

The errors endpoint response shape is unchanged. The `run_id` field is now
available on error records and **should be included** in the error response
objects:

```json
{
  "job_id": "abc-123",
```

## 5. Worker Changes

### 5.1 New parameter: `trigger`

`trigger` is passed explicitly through the enqueue path. The worker receives it
and writes it to the `job_runs` row. The worker does not infer trigger type.

**Callers:**

| Caller | Trigger value |
|---|---|
| `POST /scrape` route → `worker_pool.enqueue(...)` | `"adhoc"` |
| `SchedulerService._trigger_job` → `worker_pool.enqueue(...)` | `"scheduled"` |
| Future manual retry (not yet implemented) | `"manual"` |

**Signature changes required:**

`WorkerPool.enqueue()`:

```python
async def enqueue(
    self,
    job_id: str,
    config_yaml: str,
    priority: str = "normal",
    needs_browser: bool = False,
    *,
    run_id: str | None = None,
    trigger: str = "adhoc",
) -> QueueJobHandle:
```

`WorkerPool._run_job()`:

```python
async def _run_job(
    self,
    _ctx: dict[str, Any],
    job_id: str,
    config_yaml: str,
    run_id: str | None = None,
    *,
    needs_browser: bool = False,
    trigger: str = "adhoc",
) -> dict[str, str]:
```

`WorkerPool._execute()`:

```python
async def _execute(
    self, job_id: str, config_yaml: str,
    *, run_id: str | None = None, trigger: str = "adhoc",
) -> None:
```

`scrape_task()`:

```python
async def scrape_task(
    job_id: str,
    config_yaml: str,
    *,
    run_id: str | None = None,
    trigger: str = "adhoc",
    job_store: JobStore,
    result_store: ResultStore,
    error_store: ErrorStore,
    circuit_breaker: CircuitBreaker,
    webhook_dispatcher: WebhookDispatcher | None = None,
) -> None:
```

The `trigger` value is passed positionally through `_redis.enqueue_job()` in
`enqueue()`, matching the existing pattern for `run_id`.

### 5.2 Run lifecycle in `scrape_task()`

The following sequence replaces the current run-tracking logic:

**Step 1 — Run start (after config parse, before target processing):**

```python
import hashlib
from scrapeyard.storage.database import get_db

config_hash = hashlib.sha256(config_yaml.encode()).hexdigest()

async with get_db("jobs.db") as db:
    await db.execute(
        """INSERT INTO job_runs
           (run_id, job_id, status, trigger, config_hash, started_at)
           VALUES (?, ?, 'running', ?, ?, ?)""",
        (run_id, job_id, trigger, config_hash, started_at.isoformat()),
    )
    await db.commit()
```

This happens after the `_should_skip_delivery` check and after updating
`jobs.status = 'running'`.

**Step 2 — Error logging (throughout target processing):**

Every call to `_log_error()` now passes `run_id`. The `ErrorRecord` model gains
a `run_id: str` field. The error store's `log_error` method writes it to the
`run_id` column.

**Step 3 — Run completion (after result save, before webhook dispatch):**

```python
async with get_db("errors.db") as db:
    cursor = await db.execute(
        "SELECT COUNT(*) FROM errors WHERE run_id = ?", (run_id,)
    )
    error_count = (await cursor.fetchone())[0]

async with get_db("jobs.db") as db:
    await db.execute(
        """UPDATE job_runs
           SET status = ?, completed_at = ?, record_count = ?, error_count = ?
           WHERE run_id = ?""",
        (final_status.value, completed_at.isoformat(), len(flat_data), error_count, run_id),
    )
    await db.commit()
```

**Step 4 — Job row update (unchanged except removal of run_count/last_run_at):**

```python
job = await job_store.get_job(job_id)
job = job.model_copy(update={
    "status": final_status,
    "updated_at": completed_at,
    "current_run_id": run_id,
})
await job_store.update_job(job)
```

Note: `run_count` increment and `last_run_at` assignment are **removed** from
this update.

**Step 5 — Crash handler (existing `except Exception` block):**

The crash handler must also finalize the run row:

```python
except Exception:
    logger.exception("scrape_task crashed for job_id=%s", job_id)
    try:
        job = await job_store.get_job(job_id)
        job = job.model_copy(update={
            "status": JobStatus.failed,
            "updated_at": datetime.now(timezone.utc),
        })
        await job_store.update_job(job)

        # Finalize run row if it was created.
        if run_id is not None:
            async with get_db("jobs.db") as db:
                await db.execute(
                    """UPDATE job_runs
                       SET status = 'failed', completed_at = ?
                       WHERE run_id = ? AND status = 'running'""",
                    (datetime.now(timezone.utc).isoformat(), run_id),
                )
                await db.commit()
    except Exception:
        logger.exception("Failed to mark job %s as failed", job_id)
```

The `AND status = 'running'` guard ensures we don't overwrite a run that
already completed normally before the crash (e.g., crash during webhook
dispatch after successful result save).

### 5.3 Result formatting without the formatter system

The worker currently does:

```python
formatter = get_formatter(fmt)
formatted = formatter(job_meta, formatted_results, group_by)
```

This is replaced with inline JSON assembly. The grouping logic from
`json_fmt.py` moves directly into the worker:

```python
group_by = config.output.group_by

if group_by == GroupBy.merge:
    merged = []
    for tr in all_results:
        for item in tr.data:
            if isinstance(item, dict):
                item["_source"] = urlparse(tr.url).netloc
            merged.append(item)
    output_data = {**job_meta, "results": merged}
else:
    grouped = {}
    for tr in all_results:
        if not tr.data:
            continue
        domain = urlparse(tr.url).netloc
        grouped[domain] = {
            "status": tr.status,
            "count": len(tr.data),
            "data": tr.data,
        }
    output_data = {**job_meta, "results": grouped}

save_meta = await result_store.save_result(
    job_id,
    output_data,
    run_id=run_id,
    status=final_status.value,
    record_count=len(flat_data),
)
```

### 5.4 Result store changes

`LocalResultStore.save_result()` simplified signature:

```python
async def save_result(
    self,
    job_id: str,
    data: Any,
    *,
    run_id: str | None = None,
    status: str = "complete",
    record_count: int | None = None,
) -> SaveResultMeta:
```

**Removed parameters:** `format`, `file_contents`.

**Behavior:** Always writes `results.json` to the run directory.

`LocalResultStore.get_result()` simplified behavior:

- Always reads `results.json` from the run directory.
- Returns a structured object (not raw `Any`) containing both `run_id` and
  the parsed data, so the route handler can include `run_id` in the response
  envelope.

Suggested return type:

```python
@dataclass(frozen=True, slots=True)
class ResultPayload:
    run_id: str
    data: Any
```

The `results_meta` INSERT no longer includes a `format` column.

```python
async def get_job_runs(self, job_id: str, limit: int = 10) -> list[JobRun]:

async def get_job_run_stats(self, job_id: str) -> tuple[int, datetime | None]:
```

```sql
SELECT j.*, COUNT(r.run_id) AS run_count, MAX(r.started_at) AS last_run_at

LEFT JOIN job_runs r ON j.job_id = r.job_id
[WHERE j.project = ?]
```

## 8. Route Handler Changes

### 8.1 `GET /jobs/{job_id}` handler

- Add dependency on `SchedulerService` (already available via `get_scheduler`).
- Call `job_store.get_job_runs(job_id, limit=10)` for run history.
- Call `job_store.get_job_run_stats(job_id)` for derived `run_count` and
  `last_run_at`.
- Call `scheduler.get_next_run_time(job_id)` for `next_run_at`.
- Build expanded response per section 4.1.
